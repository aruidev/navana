
# OAuth in Navana

## Purpose

This document explains how authentication with Google and GitHub works in the project, how it is technically implemented, and what the differences are between both integrations.

The application allows three scenarios for both providers:

- Sign in with OAuth
- Link an OAuth account to an already authenticated user
- Unlink the OAuth account, as long as the user has a local password

## Involved Components

### Controllers

- `app/controller/auth/google.php`: coordinates the Google OAuth flow.
- `app/controller/auth/github.php`: coordinates the GitHub OAuth flow.

### Services

- `app/services/GoogleOAuthService.php`: builds the Google client, generates the authorization URL, and exchanges the `code` for the user's profile.
- `app/services/GithubAuthService.php`: uses `Hybridauth\Provider\GitHub` to authenticate and retrieve the user's profile.
- `app/services/UserService.php`: contains the shared business logic for login, linking, and unlinking OAuth accounts.

### Persistence

- `app/model/dao/UserDAO.php`: searches for and creates local users.
- `app/model/dao/UserOAuthAccountDAO.php`: stores the relationship between the local user and the OAuth account.
- `db_schema/Pt05_Alex_Ruiz.sql`: defines the `user_oauth_accounts` table.

### Interface

- `app/view/login.php`: displays the `Continue with Google` and `Continue with GitHub` buttons.
- `app/view/account-settings.php`: shows the linking status and allows linking or unlinking each provider.

### Configuration

Credentials are read from `environments/env.example.php` and its equivalent for each environment.

Google uses these keys:

- `oauth_google_client_id`
- `oauth_google_client_secret`
- `oauth_google_redirect_uri`

GitHub uses these keys:

- `hybrid_auth_github_client_id`
- `hybrid_auth_github_client_secret`
- `hybrid_auth_github_callback_uri`

## Data Model

The linking with external providers is not stored in the `users` table, but in a separate table:

```sql
CREATE TABLE user_oauth_accounts (
	id INT AUTO_INCREMENT PRIMARY KEY,
	user_id INT NOT NULL,
	provider VARCHAR(32) NOT NULL,
	provider_user_id VARCHAR(191) NOT NULL,
	provider_email VARCHAR(100) NULL,
	linked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
	UNIQUE KEY uq_oauth_provider_identity (provider, provider_user_id),
	UNIQUE KEY uq_oauth_user_provider (user_id, provider),
	INDEX idx_oauth_user (user_id),
	CONSTRAINT fk_oauth_user FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

This structure enforces two important rules:

- The same external identity cannot be linked to two local users
- A local user can only have one account per provider

## Shared General Flow

Although Google and GitHub use different libraries, both end up following the same business pattern inside `UserService`.

When a valid OAuth profile arrives, the system does the following:

1. Obtains `provider_user_id`, `email`, and `name`.
2. Checks if that OAuth identity is already linked in `user_oauth_accounts`.
3. If a link exists, retrieves the local user and authenticates them.
4. If no link exists, searches for a local user by email.
5. If a user with that email exists, links the OAuth account to that user.
6. If not, creates a new user with an autogenerated username and empty password.
7. Then links the OAuth account to the new user.

The above logic is duplicated for Google in `loginWithGoogle()` and reused for GitHub via `loginWithProvider()`.

## Authentication with Google

### Flow Start

The user arrives at `app/controller/auth/google.php?start=1&mode=login` from the login button, or at `?start=1&mode=link` from account settings.

The controller performs these actions:

1. Starts the session with `startSession()`.
2. Creates `UserService` and `GoogleOAuthService`.
3. Checks that Google is configured with `isConfigured()`.
4. Determines the mode: `login` or `link`.
5. If the mode is `link`, requires `$_SESSION['user_id']` to exist.
6. Generates a random `state` with `bin2hex(random_bytes(32))`.
7. Stores in session:
   - `google_oauth_state`
   - `google_oauth_mode`
   - `google_oauth_user_id`
8. Asks `GoogleOAuthService` for the authorization URL and redirects the user.

### Technical Implementation of the Service

`GoogleOAuthService` uses `Google_Client`.

In `buildClient()` it configures:

- `clientId`
- `clientSecret`
- `redirectUri`
- `accessType('online')`
- `prompt('select_account')`
- scopes `openid`, `email`, and `profile`

Then, `getAuthorizationUrl(string $state)` assigns the `state` to the client and returns `createAuthUrl()`.

### Google Callback

When Google redirects to the application, the controller expects two parameters:

- `code`
- `state`

The controller then:

1. Retrieves the expected `state` from the session.
2. Retrieves the mode and the `user_id` saved for the link flow.
3. Deletes the temporary session variables.
4. Validates the `state` with `hash_equals()`.
5. If the `state` does not match, aborts the flow for security.
6. If correct, calls `getUserProfileFromCode()`.

### Code Exchange for Profile

`getUserProfileFromCode(string $code)` does the following:

1. Builds a `Google_Client`.
2. Calls `fetchAccessTokenWithAuthCode($code)`.
3. Checks that `access_token` exists and there is no error.
4. Uses `Google_Service_Oauth2` to query `userinfo->get()`.
5. Normalizes the returned data to this format:

```php
[
	'provider' => 'google',
	'provider_user_id' => $sub,
	'email' => $email,
	'name' => $name,
]
```

If the Google identifier or email is missing, returns `null`.

### Login with Google

If the mode was `login`, the controller calls `UserService::loginWithGoogle()`.

That method:

- Searches for an existing link by provider and external identifier
- If it exists, retrieves the local user
- If not, tries to reuse an existing user by email
- If neither exists, creates a new user with an empty password
- Links the Google account to that user

If the login works, the controller:

- Regenerates the session identifier with `session_regenerate_id(true)`
- Stores `user_id`, `username`, `email`, and `is_admin`
- Executes `resetLoginAttempts()`
- Redirects to `app/view/library.php`

### Link with Google

If the mode was `link`, the controller verifies that the currently authenticated user matches the `google_oauth_user_id` saved in the session. This prevents an OAuth response from being linked to a different account.

Then it calls `UserService::linkGoogleAccount()`.

That method:

1. Validates that `provider_user_id` exists
2. Checks if that Google identity already belongs to another local user
3. If the current user already had Google linked, updates `provider_user_id` and `provider_email`
4. If not, creates a new record in `user_oauth_accounts`

### Unlink with Google

Unlink is initiated from `app/controller/auth/google.php?unlink=1`.

The controller requires an active session and calls `UserService::unlinkGoogleAccount()`.

That method only removes the link if two conditions are met:

- The user has Google linked
- The user has a local password

The second condition prevents the user from being left without any way to authenticate.

## Authentication with GitHub

### Flow Start

The user arrives at `app/controller/auth/github.php?start=1&mode=login` or `?start=1&mode=link`.

The controller does the following:

1. Starts the session.
2. Instantiates `UserService` and `GithubAuthService`.
3. Validates configuration with `isConfigured()`.
4. Determines the mode `login` or `link`.
5. If `link`, requires an authenticated user.
6. Stores in session:
   - `github_auth_mode`
   - `github_auth_user_id`

It does not manually generate a `state` in the controller. The authorization flow is delegated to HybridAuth.

### Technical Implementation of the Service

`GithubAuthService` uses `Hybridauth\Provider\GitHub`.

Its internal configuration contains:

- `callback`
- `keys.id`
- `keys.secret`
- `scope` with `user:email`

The `buildProvider()` method creates the GitHub provider instance, and `getUserProfileFromCallback()` executes authentication and then obtains the profile.

### GitHub Callback

The controller considers the callback valid if it is not in the start phase and `code` exists.

Then it retrieves from the session:

- `github_auth_mode`
- `github_auth_user_id`

Then clears those variables and calls `GithubAuthService::getUserProfileFromCallback()` inside a `try/catch`.

### Obtaining the Profile with HybridAuth

`getUserProfileFromCallback()`:

1. Creates the GitHub provider
2. Calls `authenticate()`
3. Gets the profile with `getUserProfile()`
4. Extracts `identifier`, `email`, and display name
5. Normalizes the data to this format:

```php
[
	'provider' => 'github',
	'provider_user_id' => $providerUserId,
	'email' => $email,
	'name' => $name,
]
```

If the external identifier is missing, returns `null`.

Additionally, the controller explicitly requires GitHub to provide a valid email. If the email is empty, the flow is aborted.

### Login with GitHub

For login, the controller calls `UserService::loginWithGithub()`.

This method does not repeat all the logic manually, but delegates to `loginWithProvider('github', $githubProfile)`.

That shared method performs exactly this flow:

1. Validates `provider_user_id` and `email`
2. Searches for an existing link in `user_oauth_accounts`
3. If it exists, returns the associated local user
4. If not, tries to find a local user by email
5. If found, links the GitHub account
6. If not found, generates a unique username, creates a user without a password, and links the account

If the process finishes successfully, the controller regenerates the session, fills in the user data, and redirects to `library.php`.

### Link with GitHub

In `link` mode, the controller compares the currently authenticated user with the `github_auth_user_id` saved in the session.

If they match, it calls `UserService::linkGithubAccount()`, which in turn delegates to `linkProviderAccount('github', ...)`.

The behavior is:

- If the GitHub identity is already linked to another user, it fails
- If the current user already had a GitHub link, it updates the data
- If not, it creates the linking record

### Unlink with GitHub

Unlink is triggered from `app/controller/auth/github.php?unlink=1`.

`UserService::unlinkGithubAccount()` follows the same rule as Google:

- The GitHub link must exist
- A local password must exist

Only then is the `user_oauth_accounts` record deleted.

## Technical Differences Between Google and GitHub

### 1. Library Used

- Google: manual implementation with `Google_Client` and `Google_Service_Oauth2`.
- GitHub: integration based on `Hybridauth\Provider\GitHub`.

### 2. Flow Start

- Google explicitly builds the authorization URL with `createAuthUrl()`.
- GitHub delegates the authentication flow to HybridAuth via `authenticate()`.

### 3. Callback Protection

- Google implements explicit `state` validation in the controller using `hash_equals()`.
- GitHub does not show a manual `state` in the controller; the exchange protection is encapsulated in HybridAuth.

### 4. Logic Reuse in `UserService`

- Google has a specific method `loginWithGoogle()` and another specific `linkGoogleAccount()`.
- GitHub reuses the general methods `loginWithProvider()` and `linkProviderAccount()`.

This means that the logic for Google and GitHub is functionally similar, but not abstracted at the same level of reuse.

### 5. Email Requirement

- Google already returns `null` from the service if email is missing.
- GitHub also needs email, but the strong check is also done in the controller, which rejects profiles without a valid email.

### 6. Error Handling

- Google handles errors by checking the token content and returning `null` if the exchange fails.
- GitHub encapsulates the process in `try/catch` because HybridAuth can throw exceptions during authentication.

## Interface Integration

### Login Screen

`app/view/login.php` offers two direct accesses:

- `../controller/auth/google.php?start=1&mode=login`
- `../controller/auth/github.php?start=1&mode=login`

Therefore, from the UI both providers behave as alternatives to the traditional login with username and password.

### Account Settings

`app/view/account-settings.php` queries `UserService` to know:

- If Google is linked
- If GitHub is linked
- If unlink is possible in each case

Depending on that state, it shows:

- `Link with Google` or `Link with GitHub` button
- `Unlink Google` or `Unlink GitHub` button
- Or a message indicating that unlinking is not possible until a local password is set

There is an implementation detail visible in the password change section: the informative text specifically mentions Google, although the business rule of having a local password also applies to GitHub for unlinking.

## Local User Generation

If the user authenticates with OAuth and does not yet exist in the database, the system creates a local account.

This process works as follows:

1. Takes `name` if it exists.
2. If not, uses the part before the `@` in the email.
3. Normalizes the name to lowercase.
4. Replaces invalid characters with `_`.
5. If the username already exists, adds a numeric suffix.

The resulting user is created with an empty password, which allows authentication via OAuth but prevents unlinking until a local password is set.

## Security and Design Decisions

The current implementation introduces several relevant security measures:

- Google uses a random `state` to reduce CSRF risk in the callback.
- After a successful login, `session_regenerate_id(true)` is executed to reduce session fixation risk.
- The `link` mode saves the `user_id` in the session and checks it in the callback to avoid linking the OAuth identity to a different session.
- The `user_oauth_accounts` table prevents duplicate links with unique constraints.
- Unlinking is not allowed if the user does not have a local password, preventing the account from becoming inaccessible.
- All persistence is done with PDO and prepared statements through the DAOs.

## Current Limitations

The implementation works, but there are some technical aspects to consider:

- The Google logic is partially duplicated compared to the general abstraction used by GitHub.
- Callback security validation is more explicit in Google than in GitHub at the controller level.
- Users created from OAuth initially have no local password.
- The project documentation must assume the current real behavior, not a completely homogeneous architecture between providers.

## Comparative Summary

| Aspect | Google | GitHub |
| --- | --- | --- |
| Library | `Google_Client` | `Hybridauth\Provider\GitHub` |
| Flow start | Manual URL with `createAuthUrl()` | HybridAuth's `authenticate()` |
| Callback | `code` + `state` | `code` managed by HybridAuth |
| Visible CSRF validation | Yes, with `state` | Not visible in the controller |
| Main scope | `openid email profile` | `user:email` |
| Business service | `loginWithGoogle()` | `loginWithGithub()` -> `loginWithProvider()` |
| Manual link | `linkGoogleAccount()` | `linkGithubAccount()` -> `linkProviderAccount()` |
| Persistence | `user_oauth_accounts` | `user_oauth_accounts` |
| Unlink allowed | Only with local password | Only with local password |

## Conclusion

Navana's OAuth authentication is built on a simple architecture consistent with the project: controllers handle HTTP and session, services encapsulate communication with providers and business logic, and DAOs persist the links in the database.

Google and GitHub offer the same functional result for the user, but with clear technical differences:

- Google uses a more manual and explicit OAuth flow.
- GitHub relies on HybridAuth.
- Both share the same strategy for associating with local users and the same security rule for unlinking.

Overall, the system allows social login, association of existing accounts, and protection against duplicates or accidental access lockouts.
